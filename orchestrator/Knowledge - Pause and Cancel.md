# Project Pause and Cancellation

## Project Pause Process

### When to Pause a Project
A project may be paused due to:
- External dependencies not met
- Resource constraints
- Strategic priority changes
- Funding issues
- Other circumstances requiring a hold

---

### Request to Pause

#### Step 1: Submit Decision Request
The Project Sponsor must submit a Decision Request for approval using the secretarial process.

**Required Information:**
- **Reason for pause**: Detailed explanation of circumstances
- **Risks**: Risks associated with pausing the project
- **Duration**: Proposed pause length
- **Reassessment date**: Anticipated date to evaluate project status

#### Step 2: Generate Change Request
Upon PMB approval of the Decision Request, generate a CR to:
- Align current fiscal year costs to actuals to date
- Update current fiscal year forecast to reflect the pause

---

### During Pause Period

#### Project Team Must:
1. **Notify stakeholders** - All team members and stakeholders informed of paused status
2. **Update EPS** - Set Overall Status to "paused"
3. **Maintain artifacts** - Ensure EPS contains last approved and signed artifacts including:
   - Decision Request to Pause
   - Signed CR to align costs
4. **Periodic reviews** - Conduct reviews to assess feasibility of resuming

#### Oversight During Pause
- Paused projects discussed with PMDB ADM at Sponsor meetings
- Shared with PMB every **6 months** starting from initial pause date

---

### Reinstatement Process

#### To Reinstate a Paused Project:
1. **Evaluation**: Conduct assessment to determine if pause conditions are resolved/mitigated
2. **Viability analysis**: Updated analysis considering changes in:
   - Strategic priorities
   - Resources
   - Funding availability
3. **PMB Approval**: Seek approval via secretarial Decision Request
4. **Scope/cost/schedule changes**: Any adjustments require DG-approved CR
5. **10% threshold**: Changes >= 10% require PMB gate update

---

## Project Cancellation Process

### Request to Cancel

#### Step 1: Submit Decision Request
Project Sponsor submits Decision Request to PMB using secretarial process.

**Required Information:**
- **Reason for cancellation**: Detailed circumstances
- **Dependencies**: Dependencies on other projects
- **Risks**: Risks associated with cancelling
- **Cancellation date**: Proposed official cancellation date
- **Closeout steps**: Necessary steps to close out project

#### Step 2: Generate DG-Approved CR
Submit DG-approved Change Request to:
- Align project costs to actuals
- Ensure all actuals have gone through Financial System
- No outstanding transactions

---

### After Cancellation Approved

#### Project Team Must:
1. **Notify stakeholders** - All team members informed of cancellation
2. **Update EPS** - Set Overall Status to "cancelled"
3. **Maintain artifacts** - Ensure EPS contains:
   - Decision Request to Cancel
   - Signed CR to close financials
4. **Final review** - Ensure all project activities properly closed out

---

### Finalizing Cancellation

#### Required Actions:
1. Confirm all conditions addressed
2. Close project IO
3. Sign project change request

#### PMCoE Responsibilities:
- Update Callipers to inform TBS of cancellation

---

## Key Differences: Pause vs. Cancel

| Aspect | Pause | Cancel |
|--------|-------|--------|
| **Status in EPS** | Paused | Cancelled |
| **IO Code** | Can be reopened upon reinstatement | Closed |
| **Funding** | Can be re-allocated upon reinstatement | Funds decommitted |
| **Governance** | 6-month review with PMB | Final closure |
| **Team** | Retained (may be reassigned) | Released |

---

## Financial Considerations

### During Pause
- CR aligns costs to actuals
- IO code may be closed
- Unused funds can be decommitted

### Upon Cancellation
- All actuals must be processed
- No outstanding transactions
- Final bill sent to customer (Customer-Led projects)
- Decommit unused funds
